# OpenSearch 3.6 Cluster — Startup Procedure

This document describes the complete, ordered procedure to bring the 5-node cluster up from zero (first time) or after a full data reset. It also captures every quirk and trap discovered during real operation so nothing gets lost between sessions.

---

## Directory layout — read this first

```txt
where-you-put-it/                         ← parent dir
└── OpenSearch-3.6/                       ← WORKING DIRECTORY for this cluster
    ├── .env
    ├── docker-compose.yml
    ├── opensearch_installer_vars.cfg
    ├── opensearch_local_certificates_creator.sh
    ├── restart-to-clear-cluster.sh
    ├── initial_api_calls.sh
    └── assets/
        ├── ssl/                          ← TLS certificates (generated by the creator script)
        ├── opensearch/
        │   ├── config/os0{1..5}/         ← per-node opensearch.yml, jvm.options, log4j2.properties
        │   │   └── opensearch-security/  ← only present under os01
        │   ├── data/os0{1..5}data/       ← bind-mounted data directories
        │   └── performance-analyzer/
        └── dashboards/
            └── opensearch_dashboards.yml
```

> **Critical:** Always `cd` into `OpenSearch-3.6/` before running any `docker compose` command.
> The parent directory also contains a `docker-compose.yml` that starts a completely different
> 2-node cluster. Running compose from the parent directory will start the wrong project and
> leave stray containers that block ports 9200 and 5601.

## Prerequisites

| Requirement | Notes |
|---|---|
| Docker + Docker Compose plugin | `docker compose version` must work |
| OpenSSL available either in `PATH` or from a standard Windows install | The PowerShell certificate creator auto-detects `openssl.exe` from Git for Windows (`Git\\usr\\bin` or `Git\\mingw64\\bin`) and common OpenSSL installs |
| `vm.max_map_count ≥ 262144` on the host | OpenSearch will not start without this; set with `sudo sysctl -w vm.max_map_count=262144` or persist in `/etc/sysctl.conf` |
| Free ports 9200, 9600, 5601 | Verify with `ss -tlnp | grep -E '9200|9600|5601'` |

## Scenario A — Absolute first-time setup (no data, no certs)

### Step 1 — Enter the working directory

```bash
cd /path/to/where-you-put-it/OpenSearch-3.6
```

All subsequent commands assume this as the current working directory.

### Step 2 — Verify (or create) the `.env` file

The file must export:

```bash
OPEN_SEARCH_VERSION=3.6.0
OPENSEARCH_INITIAL_ADMIN_PASSWORD="test@Cici24#ANA"
ES_PORT=9200
OS_CLUSTER_NAME="opensearch"
OS_SEARCH_SNAPSHOT_SIZE="2gb"
```

The `OPENSEARCH_INITIAL_ADMIN_PASSWORD` must satisfy OpenSearch 2.12+ strong-password policy: minimum 8 characters, at least one uppercase, one lowercase, one digit, one special character.

### Step 3 — Generate TLS certificates and write node configs

```bash
bash opensearch_local_certificates_creator.sh
```

This script:

1. Creates a self-signed Root CA (`assets/ssl/root-ca.pem` + key);
2. Creates a TLS admin certificate (`assets/ssl/admin.pem` + key);
3. Creates per-node certificates for os01–os05, client, dashboards;
4. Generates a new **16-character compliance salt** and writes it to `plugins.security.compliance.salt` in all 5 `opensearch.yml` files;
5. Generates a new **32-hex-char SQL datasource master key** and writes it to `plugins.query.datasources.encryption.masterkey` in all 5 `opensearch.yml` files;
6. Sets secure file permissions (certs: 600, config dirs: 700, config files: 600).

> **Warning — re-running this script on a live cluster:**
> Generating a new SQL master key on a cluster that already has stored datasource credentials
> will make those credentials unreadable. Only run this script on a fresh/reset cluster.

> **Save the output values.** The script prints the compliance salt and master key to stdout.
> Record them securely — they are needed if you ever need to restore from backup.

### Step 4 — Build the custom os01 image

Creation of the container `os01` uses a local `Dockerfile` instead of the stock image:

```bash
docker compose build os01
```

This only needs to be done once (or when the Dockerfile changes). The other nodes (os02–os05, dashboards) use the official `opensearchproject/opensearch:3.6.0` image pulled automatically.

### Step 5 — Start the cluster

```bash
docker compose up
```

Or in the background:

```bash
docker compose up -d
```

With live log tee for diagnostics:

```bash
docker compose up 2>&1 | tee /tmp/opensearch-startup.log
```

Expected boot sequence:

1. os01 starts first and begins cluster formation;
2. os02–os05 join. The cluster reaches quorum once os01 and os02 are both up;
3. os01 becomes cluster-manager (it holds the `cluster_manager` role exclusively);
4. The Security plugin initialises the `.opendistro_security` index on os01 from the YAML files in `assets/opensearch/config/os01/opensearch-security/` because `plugins.security.allow_default_init_securityindex=true` is set.
5. dashboards connects to os01 and os02.

Typical time to a healthy cluster: **60–120 seconds** depending on hardware.

### Step 6 — Verify the cluster

```bash
# Health check — should return "green"
docker exec os01 curl -sk https://os01:9200/_cluster/health \
  -u "admin:${OPENSEARCH_INITIAL_ADMIN_PASSWORD}" | python3 -m json.tool

# Node list
docker exec os01 curl -sk "https://os01:9200/_cat/nodes?v" \
  -u "admin:${OPENSEARCH_INITIAL_ADMIN_PASSWORD}"
```

Expected node table:

```
ip         heap%  ram%  cpu  ...  node.role  cm    name
172.x.x.x    xx   ...               s         -    os05   (search node)
172.x.x.x    xx   ...               m         *    os01   (cluster-manager, elected)
172.x.x.x    xx   ...               di         -    os04   (data+ingest)
172.x.x.x    xx   ...               dim        -    os02   (cluster-manager-eligible+data+ingest)
172.x.x.x    xx   ...               di         -    os03   (data+ingest)
```

## Scenario B — Full reset and restart (data wipe)

Use this procedure whenever you need to start completely clean — for example after a failed
configuration change or to reproduce a clean-state issue.

### Step 1 — Run the reset script

```bash
cd /path/to/cluster-opensearch/OpenSearch-3.6
bash restart-to-clear-cluster.sh
```

This script:

- Stops **all** running Docker containers (`docker stop $(docker ps -aq)`).
- Removes **all** containers (`docker container rm -f ...`).
- Removes dangling volumes.
- Removes the custom `cluster-opensearch-os01:latest` image.
- Wipes `assets/opensearch/data/os0{1..5}data/*`.

> **Note:** The script stops *all* Docker containers on the machine, not just this project's.
> If you run other containers, stop only the compose project instead:
> ```bash
> docker compose down
> rm -rf assets/opensearch/data/os0{1..5}data/*
> ```

Non-fatal errors you will see (safe to ignore):

- `No such volume` — no named Docker volumes exist (we use bind mounts).
- `Error response from daemon: No such image: cluster-opensearch-os01:latest` — image was
  already removed or never built in the first session.

### Step 2 — Regenerate certificates

```bash
bash opensearch_local_certificates_creator.sh
```

Because the data directory was wiped, new certificates are fine. The old cert files in
`assets/ssl/` are overwritten with freshly generated ones. See Step 3 in Scenario A for what
this script does and its warnings.

### Step 3 — Rebuild the os01 image

```bash
docker compose build os01
```

The previous image was removed in Step 1. Skip this only if you know the image is still present
(`docker images | grep cluster-opensearch-os01`).

### Step 4 — Start the cluster

Same as Scenario A, Step 5.

```bash
docker compose up 2>&1 | tee /tmp/opensearch-startup.log
```

### Step 5 — Wait for os01 to become healthy, then push security config

**This is the most common gotcha after a data wipe.**

After a full data reset, `allow_default_init_securityindex=true` should cause the Security
plugin to auto-initialise the `.opendistro_security` index from the YAML files on disk.
However, the auto-init can race with cluster formation and sometimes leave the index empty.

**Symptom:** `curl` returns `401 Unauthorized` even with correct credentials.

**Check:**

```bash
docker exec os01 curl -sk https://os01:9200/_cluster/health \
  -u "admin:${OPENSEARCH_INITIAL_ADMIN_PASSWORD}"
```

If you get `Unauthorized`, the security index needs to be pushed manually:

```bash
docker exec os01 bash -c '
  CONF=/usr/share/opensearch/config/opensearch-security
  /usr/share/opensearch/plugins/opensearch-security/tools/securityadmin.sh \
    -cd $CONF \
    -h os01 -p 9200 \
    -cert /usr/share/opensearch/config/admin.pem \
    -key  /usr/share/opensearch/config/admin-key.pem \
    -cacert /usr/share/opensearch/config/root-ca.pem \
    -nhnv
'
```

> **OpenSearch 3.x change:** `securityadmin.sh` now connects over **REST on port 9200**, not
> transport port 9300. The `-p 9200` flag is mandatory.

> **Certificate paths:** Certs are mounted directly into
> `/usr/share/opensearch/config/` — there is **no** `tls/` subdirectory.

On success, the output will say `Done with success` for each of the 9 config types
(config, roles, rolesmapping, internalusers, actiongroups, tenants, nodesdn, whitelist, audit)
across all 5 nodes.

### Step 6 — Start the Dashboards container separately (if not started with compose up)

```bash
docker compose up -d dashboards
```

Dashboards can be started any time after os01 is healthy. It uses os01 and os02 as backend
hosts (`OPENSEARCH_HOSTS=["https://os01:9200","https://os02:9200"]`).

### Step 7 — Verify

Same as Scenario A, Step 6.

## Known quirks and traps

### Quirk 1 — Wrong working directory starts a different cluster

The parent directory `cluster-opensearch/` contains its own `docker-compose.yml` that launches
a 2-node default cluster (`opensearch-node1`, `opensearch-node2`). If any `docker compose`
command is run from the parent directory instead of `OpenSearch-3.6/`, the wrong cluster starts.

Those nodes fail immediately with:
```
No custom admin password found. Please set password using the OPENSEARCH_INITIAL_ADMIN_PASSWORD environment variable.
```

**Fix:** Stop and remove them, then start from the correct directory.

```bash
docker stop opensearch-node1 opensearch-node2 opensearch-dashboards
docker rm   opensearch-node1 opensearch-node2 opensearch-dashboards
cd OpenSearch-3.6/
docker compose up
```

### Quirk 2 — Port 5601 held by a stray dashboards container

If a previous run left a dashboards container from the wrong project, port 5601 is blocked.
The correct `dashboards` container will fail to start with a port-binding error.

**Fix:**

```bash
docker ps -a --filter "publish=5601"
docker rm -f <container-id>
docker compose up -d dashboards
```

### Quirk 3 — `securityadmin.sh` must be run after every clean data reset

Even with `allow_default_init_securityindex=true`, the security index auto-init sometimes does
not complete reliably on a totally fresh data directory. Always test with a `curl` health check
before assuming the cluster is ready. If you get `401 Unauthorized`, run `securityadmin.sh`
as described in Scenario B, Step 5.

### Quirk 4 — E3: `.opendistro_security` missing on os02–os05 at boot (transient)

During the first ~30 seconds, os02–os05 log:
```
[WARN] index '.opendistro_security' missing
```
This is transient — they are waiting for os01 to finish initialising the security index and
replicate it to the other nodes. The warning disappears once replication completes. No action
needed.

### Quirk 5 — Compliance salt and SQL master key must be identical on all nodes

Both values are written to all 5 `opensearch.yml` files by `opensearch_local_certificates_creator.sh`.
If you manually edit a single node's `opensearch.yml`, ensure you replicate the change to
all other nodes. A mismatch causes the Security plugin to fail with
`SettingsException: unknown setting` or silent decryption errors.

### Quirk 6 — os01 uses a custom Docker image; os02–os05 use the stock image

os01 is built from `assets/opensearch/Dockerfile`. After `restart-to-clear-cluster.sh` the
built image is deleted. You must run `docker compose build os01` before the next `docker compose up`.

If you skip this step, Docker will fail to find the image and os01 will not start.

### Quirk 7 — `restart-to-clear-cluster.sh` stops all containers on the host

The script calls `docker stop $(docker ps -aq)` which targets every running container, not just
this project. On a shared machine, use targeted commands instead:

```bash
docker compose down
rm -rf assets/opensearch/data/os0{1..5}data/*
```

### Quirk 8 — Auth failure logs: `Authentication finally failed for null`

After a clean start you may see this warning in os01's logs every ~5 seconds:
```
[WARN][...] Authentication finally failed for null from 172.x.x.x:XXXXX
```
`null` means the request carried no `Authorization` header. The source IP is typically the
Dashboards container running its healthcheck against the cluster before it has received
credentials. This is a transient startup warning and stops once Dashboards is fully
initialised. It can also appear from the os01 healthcheck (`/cat/nodes`) if the container
healthcheck fires before the Security index is ready.

## Quick-reference command cheatsheet

```bash
# --- Reset ---
cd /path/to/cluster-opensearch/OpenSearch-3.6
bash restart-to-clear-cluster.sh
bash opensearch_local_certificates_creator.sh
docker compose build os01

# --- Start ---
docker compose up 2>&1 | tee /tmp/opensearch-startup.log

# --- Health check ---
docker exec os01 curl -sk https://os01:9200/_cluster/health \
  -u "admin:test@Cici24#ANA" | python3 -m json.tool

# --- Node list ---
docker exec os01 curl -sk "https://os01:9200/_cat/nodes?v" \
  -u "admin:test@Cici24#ANA"

# --- Push security config manually (if 401 after clean start) ---
docker exec os01 bash -c '
  CONF=/usr/share/opensearch/config/opensearch-security
  /usr/share/opensearch/plugins/opensearch-security/tools/securityadmin.sh \
    -cd $CONF -h os01 -p 9200 \
    -cert /usr/share/opensearch/config/admin.pem \
    -key  /usr/share/opensearch/config/admin-key.pem \
    -cacert /usr/share/opensearch/config/root-ca.pem \
    -nhnv
'

# --- Check logs ---
docker logs os01 2>&1 | grep -E "WARN|ERROR" | \
  grep -v "sun.misc.Unsafe\|objectFieldOffset\|bc-fips\|Restricted method\|incubator\|arrow.memory\|AUTHENTICATED\|GRANTED_PRIVILEGES"

# --- Stop cluster (preserving data) ---
docker compose down

# --- Stop and wipe everything ---
bash restart-to-clear-cluster.sh
```

## Environment variables reference

| Variable | Where set | Purpose |
|---|---|---|
| `OPEN_SEARCH_VERSION` | `.env` | Docker image tag for all nodes and dashboards |
| `OPENSEARCH_INITIAL_ADMIN_PASSWORD` | `.env` | Admin user password; must meet strong-password policy |
| `ES_PORT` | `.env` | REST API host port (default 9200) |
| `OS_CLUSTER_NAME` | `.env` | Logical cluster name (`cluster.name`) |
| `OS_SEARCH_SNAPSHOT_SIZE` | `.env` | Search node cache size for os05 |
| `CERT_DN` | `opensearch_installer_vars.cfg` | Distinguished name prefix for certificates |
| `LOCAL_ROOT_CA` | `opensearch_installer_vars.cfg` | CN for the Root CA cert |
| `ADMIN_CA` | `opensearch_installer_vars.cfg` | CN / filename for the admin cert |
| `OS_CERTS_PATH` | `opensearch_installer_vars.cfg` | Path where certs are written (`./assets/ssl`) |
